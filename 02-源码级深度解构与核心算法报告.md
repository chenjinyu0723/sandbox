## 2. 核心编译流水线深度解剖

### 2.1 流水线总览（`src/compiler/index.ts`）

整个编译流水线由 `runCompilePipeline()` 函数编排，该函数在**项目级文件锁**保护下运行：

```
锁获取 → 日志恢复 → 变更检测 → 分类 → Phase 1 并行提取 → Phase 2 并行生成 → 种子页 → 解析 → 索引 → 锁释放
```

**关键设计决策：**

1. **文件锁 (`acquireLock` / `releaseLock`)** 保证同一时刻只有一个编译进程运行。锁文件位于 `.llmwiki/lock`。

2. **WAL 日志恢复 (`recoverJournalBeforeCompile`)** 在编译开始时首先重放任何未完成的日志——如果上一次编译崩溃，这保证了文件系统的一致性。

3. **增量编译**：通过 SHA-256 哈希比较 `sources/` 目录中的文件与 `.llmwiki/state.json` 中的记录，只处理新增/变更的源文件。

### 2.2 变更检测引擎（`src/compiler/hasher.ts` + `src/compiler/delta.ts`）

```typescript
// 变更检测的核心逻辑（简化伪代码）

async function detectChanges(root: string, state: WikiState): Promise<SourceChange[]> {
  const changes: SourceChange[] = [];
  
  for (const filename of await listSourceFiles(root)) {
    const currentHash = await hashFile(path.join(root, SOURCES_DIR, filename));
    const previousHash = state.sources[filename]?.hash;
    
    if (!previousHash) {
      changes.push({ file: filename, status: "new" });          // 新文件
    } else if (currentHash !== previousHash) {
      changes.push({ file: filename, status: "changed" });       // 内容变更
    } else {
      changes.push({ file: filename, status: "unchanged" });     // 未变更
    }
  }
  
  // 检测已删除的源文件
  for (const filename of Object.keys(state.sources)) {
    if (!existsSync(path.join(root, SOURCES_DIR, filename))) {
      changes.push({ file: filename, status: "deleted" });
    }
  }
  
  return changes;
}
```

**状态持久化** (`CompileStateDraft`):
- 整个编译运行期间维护 `.llmwiki/state.json` 的一个**内存草稿**
- 所有状态读取和写入都经过这个草稿
- **唯一的一次持久化刷新**发生在 `finalizeWiki()` 完成解析阶段之后
- 这保证了编译的原子性——如果中途崩溃，状态文件不会被部分更新

### 2.3 概念提取阶段（Phase 1: `src/compiler/extraction-phase.ts`）

```typescript
// Phase 1 核心流程
export async function runExtractionPhases(
  root: string,
  toCompile: SourceChange[],     // 直接变更的源
  state: WikiState,
  allChanges: SourceChange[],
  concurrency: number,
): Promise<ExtractionResult[]> {
  const limit = pLimit(concurrency);  // 并发上限

  // Step 1: 并行提取直接变更的源
  const extractions = await extractSourcesLimited(
    root, toCompile.map(c => c.file), limit
  );

  // Step 2: 发现受影响的未变更源（概念重叠）
  const lateAffected = findLateAffectedSources(extractions, state, allChanges);
  
  // Step 3: 并行提取受影响的源
  const lateExtractions = await extractSourcesLimited(root, lateAffected, limit);
  extractions.push(...lateExtractions);

  return extractions;
}

// 单个源的提取
async function extractForSource(root: string, sourceFile: string): Promise<ExtractionResult> {
  const sourcePath = path.join(root, SOURCES_DIR, sourceFile);
  const sourceContent = await readFile(sourcePath, "utf-8");
  
  // 读取现有 Wiki 索引作为去重上下文
  const existingIndex = await readConfinedExtractionIndex(root);
  
  // 调用 LLM 提取概念（使用 tool_use 模式）
  const concepts = await extractConcepts(sourceContent, existingIndex);
  
  return { sourceFile, sourcePath, sourceContent, concepts };
}
```

### 2.4 页面生成阶段（Phase 2: `src/compiler/index.ts` → `generatePagesPhase()`）

```typescript
async function generatePagesPhase(
  root: string,
  extractions: ExtractionResult[],
  frozenSlugs: Set<string>,     // 冻结的 slug（避免重复编译失败的页面）
  schema: SchemaConfig,
  options: CompileOptions,
  policy: ReviewPolicy,
  concurrency: number,
): Promise<PageGenerationResult> {
  // Step 1: 合并同名概念
  const merged = mergeExtractions(extractions, frozenSlugs);
  
  // Step 2: 如果启用审查模式，预先构建源状态快照
  const sourceStates = (options.review || !isPolicyOff(policy))
    ? await buildExtractionSourceStates(root, extractions)
    : {};
    
  // Step 3: 并行生成合并页面
  const limit = pLimit(concurrency);
  const outcomes = await Promise.all(
    merged.map(entry => limit(async () => {
      return await generateMergedPage(root, entry, schema, options, sourceStates, policy);
    }))
  );

  // Step 4: 分离 liveWrite（直接写入）和 candidate（审查暂存）
  const { writtenPages, batchErrors } = await commitLivePageWrites(root, outcomes);
  
  return { pages, writtenPages, errors, candidates, review };
}
```

**关键行为：**
- `mergeExtractions`：当多个源文件提取出同名概念时，合并为一个合并概念 → 产生一个合并页面
- `generateMergedPage`：调用 LLM 生成包含完整 frontmatter、正文和引文溯源的 Markdown 页面
- `commitLivePageWrites`：将所有 `liveWrite` 页面作为**单个日志批次**原子提交——要么全部成功，要么全部回滚

### 2.5 关键数据流：源文件到页面的完整路径

```
sources/rfc4271.md
    │
    ├─ SHA-256: "a1b2c3..." (与 state.json 比较)
    │   └─ status: "new" → 进入 toCompile
    │
    ├─ Phase 1: extractForSource()
    │   ├─ readFile → sourceContent
    │   ├─ readConfinedExtractionIndex → existingIndex (去重上下文)
    │   ├─ callClaude(extract_concepts tool)
    │   │   └─ 返回: [{ concept: "BGP Path Selection", summary: "...", is_new: true, ... },
    │   │            { concept: "LOCAL_PREF Attribute", summary: "...", is_new: true, ... }]
    │   └─ ExtractionResult { sourceFile, sourcePath, sourceContent, concepts }
    │
    ├─ mergeExtractions(所有 ExtractionResults, frozenSlugs)
    │   └─ MergedConcept { 
    │        slug: "bgp-path-selection",
```

## 3. 信任系统完整解析

### 3.1 信任系统的设计哲学

信任系统是 llm-wiki-compiler 最具创新性的工程架构。它的核心原则是：

> **没有任何代码路径可以绕过检查直接写入 `wiki/` 目录。**

所有的写入——无论是编译生成、审查批准、OKF 导入还是 SDK 调用——都通过同一个写入缝：`src/trust/planner.ts → planPageMutation()`。

### 3.2 写入规划器（`src/trust/planner.ts`）

```typescript
// planPageMutation 的核心逻辑
export async function planPageMutation(input: PlanPageInput): Promise<PlanResult> {
  const { target } = input;

  if (target.kind === "entity") {
    // 类型化实体页面（CLP 声明的类型）
    const targetPath = ["wiki", target.entityType, `${target.slug}.md`].join(path.sep);
    return planPageWith(
      { ...input, targetPath },
      checkIdentitySafe(target.entityType, target.slug),   // slug-safe 检查
      () => ({ entityType: target.entityType, slug: target.slug, id: entityId(...) }),
    );
  }

  // 默认页面（concepts/queries）
  const targetPath = ["wiki", target.directory, `${target.slug}.md`].join(path.sep);
  return planPageWith(
    { ...input, targetPath },
    checkDefaultIdentitySafe(target.directory, target.slug),  // filename-safe 检查
    () => ({ directory: target.directory, slug: target.slug }),
  );
}
```

**关键安全设计：路径拼接**

注意 `targetPath` 使用 `[..., ...].join(path.sep)` 而非 `path.join()`。**这是故意的**：`path.join()` 会静默地将 `../escape` 规范化为合法路径，从而隐藏路径遍历攻击。使用 `[].join()` 保留原始的 `..` 序列，让后续的 `confineUnderRoot()` 能够检测到并拒绝它。

### 3.3 强制核心检查（`src/trust/checks.ts`）

CLP Invariant 3 规定**四项不可禁用的强制检查**。任何 profile 都无法绕过它们：

```typescript
// 四项强制检查（按评估顺序）
export const mandatoryPageChecks = [
  checkPathConfinement,     // ① 路径限制
  checkTargetCollision,     // ② 冲突检测
  checkResourceLimit,       // ③ 资源上限
  checkFrontmatter,         // ④ Frontmatter 解析
];

// 保证不可绕过：runMandatoryPageChecks 只接受 PageWriteContext，
// 无 profile/predicate/skip 参数
export async function runMandatoryPageChecks(ctx: PageWriteContext): Promise<TrustCheckResult[]> {
  return Promise.all(mandatoryPageChecks.map(check => check(ctx)));
}
```

**每项检查的详细逻辑：**

| 检查 | 算法 | 拒绝条件 |
|------|------|---------|
| `checkPathConfinement` | `confineUnderRoot(targetPath, root)` | 路径逃逸项目根目录（含 symlink 祖先逃逸） |
| `checkTargetCollision` | `lstat(confined_abs_path)` | 目标已存在 且 `allowOverwrite=false` |
| `checkResourceLimit` | `body.length > cap` | 超出字符上限（编译页面用 `GENERATED_PAGE_MAX_CHARS`，其他用 `MAX_SOURCE_CHARS`） |
| `checkFrontmatter` | `parseFrontmatterStatus(body).malformedFrontmatter` | YAML frontmatter 存在但格式错误 |

### 3.4 决策组合引擎（`src/trust/decision.ts`）

```typescript
// composeTrustDecision 的逻辑
function composeTrustDecision(checks: TrustCheckResult[], options): TrustDecision {
  // 任何一个 block → deny
  if (checks.some(c => c.verdict === "block")) return "deny";
  
  // 任何一个 warn → allow-with-warning
  if (checks.some(c => c.verdict === "warn")) return "allow-with-warning";
  
  // 全部 pass → allow
  return "allow";
}
```

通过后，`planPageWith()` 构建一个 `PagePlannedMutation`：

```typescript
// 仅在允许/允许带警告时构建变异
if (!LIVE_WRITE_DECISIONS.has(decision)) {
  return { planned: [], decision, checks };
}

const exists = await targetAlreadyExists(root, targetPath);
const mutation: PagePlannedMutation = {
  kind: "page",
  operation: exists ? "update" : "create",
  target: buildTarget(),        // EntityRef 或 RawPageRef
  body: input.body,
  provenance: { origin, decision, reviewRouted },
};
return { planned: [mutation], decision, checks };
```

### 3.5 WAL 日志系统（`src/trust/journal.ts` + `journal-recovery.ts`）

日志系统提供崩溃恢复保证：

```
编译开始 → recoverJournalBeforeCompile()
              │
              ├─ 读取 .llmwiki/journal（如果存在）
              ├─ 检查是否是不完整的批次（上次编译崩溃）
              ├─ 如果是：回滚到批次前的状态
              └─ 标记为 "safe"
              │
编译过程 → 每次写入操作通过 executor
              │
              ├─ executor 打开日志批次
              ├─ 原子写入所有变异
              ├─ 关闭日志批次（标记完整）
              └─ 如果中途崩溃 → 下次 recoverJournalBeforeCompile 会回滚
              │
编译结束 → 日志已安全关闭
```

**设计保证：**
- 单个日志批次是原子的——要么所有页面都写入，要么全部回滚
- 空批次不打开日志（避免伪造的"编译不完整"恢复窗口）
- `applyCompilePageWritesLocked` 使用无锁的 executor 核心（编译已持有锁，避免死锁）

## 4. CLP（Configurable Lifecycle Profiles）引擎

### 4.1 Profile 生命周期

```
template init       profile.json      loadProfile()        validate()
     │                  │                  │                   │
     ▼                  ▼                  ▼                   ▼
  模板安装     →    声明式合约     →    运行时加载     →    Fail-closed 验证
 (autosci/          (entities/         (ProfilePack       (AJV JSON Schema
  newsroom/          relations/         类型, 包括           + 自定义
  自定义)            lifecycle/         EntityId          业务逻辑)
                     workflows/         品牌类型)
                     artifacts/
                     gates/...)
```

### 4.2 Profile 的运行时强制执行点

Profile 不是"建议"，而是"合约"。在以下所有写入路径上强制执行：

1. **页面写入** → `planPageMutation()` 检查 field contracts, relation requirements
2. **生命周期转换** → `transitionLifecycle()` 验证状态机
3. **关系写入** → `appendRelation()` 检查 relation cardinality
4. **Artifact 写入** → `writeArtifact()` 验证 artifact types
5. **OKF 导入** → 导入的页面必须符合 profile
6. **Workflow 执行** → workflow actions 必须符合声明的 gates

### 4.3 EntityId 品牌类型

```typescript
// src/profile/identity.ts
// EntityId 是一个品牌化的字符串类型，防止与普通字符串混淆
type EntityId = string & { readonly __brand: "EntityId" };

// 只能在通过身份验证后铸造
function entityId(entityType: string, slug: string): EntityId {
  return `${entityType}/${slug}` as EntityId;
}

// Slug-safe 验证
function isSlugSafe(value: string): boolean {
  return /^[a-z0-9]([a-z0-9-]*[a-z0-9])?$/.test(value);
}
```

### 5.3 安全模型

```
MCP 安全性分层：

Layer 1: Transport — stdio (父进程通信，无网络暴露)
Layer 2: 只读工具 — 永远不需要 provider 凭证
Layer 3: LLM 工具 — 在调用时验证 provider 凭证 (ensureProviderAvailable)
Layer 4: 写入限制 — MCP 动作硬限制为 staged-write，不能执行信任写入

Layer 6: 路径限制 — source confinement 防止路径遍历
```
