## 1. 基础使用场景

### 1.1 Quick Start 工作流

```
第 1 步: 创建项目
  → 启动应用 → 点击 "New Project"
  → 选择场景模板: Research / Reading / Personal Growth / Business / General
  → 输入项目名称和路径
  → 模板会预配置 purpose.md 和 schema.md

第 2 步: 配置 LLM
  → Settings → LLM Provider
  → 选择 Provider + 填入 API Key + 选择 Model
  → 推荐: Claude Sonnet 4 (Ingest) + GPT-4o (Chat)

第 3 步: 导入文档
  → Sources → Import → 选择 PDF/DOCX/MD 文件或整个文件夹
  → 观察 Activity Panel — LLM 自动构建 Wiki 页面
  → 等待队列清空

第 4 步: 查询知识库
  → Chat → 输入问题
  → Agent 自动检索相关 Wiki 页面，引用来源

第 5 步: 维护
  → 定期运行 Lint (发现孤立页面、断裂链接等)
  → 检查 Review 面板 (处理 LLM 标记的待审项)
```

### 1.2 场景一：个人研究知识库

**适用人群：** 研究人员、博士生、分析师

**配置：**
```markdown
# purpose.md (Research 模板)
## Goals
- 跟踪 [研究领域] 的最新进展
- 建立关键论文和方法的引用网络
- 发现研究空白和交叉领域

## Key Questions
1. [领域] 的 SOTA 方法是什么？
2. 不同方法的 trade-off 是什么？
3. 还有哪些开放问题？

## Research Scope
- 主要关注: [领域 A], [领域 B]
- 辅助关注: [领域 C]
```

**典型工作流：**
1. 每次发现新论文 → 下载 PDF → 拖入 Sources
2. LLM 自动提取实体（方法、数据集、作者）、概念（理论框架）和关系
3. 在 Chat 中提问："比较方法 A 和方法 B 的性能"
4. 查看 Knowledge Graph 发现跨领域连接

### 1.3 场景二：团队文档 Wiki

**适用人群：** 工程团队、产品团队

**配置：**
```markdown
# purpose.md (Business 模板)
## Goals
- 将产品文档、会议记录、技术规范转化为可查询的知识库
- 新成员 onboarding 的知识入口
- 减少重复讨论

## Key Questions
1. 为什么做了 X 决策？
2. 系统 Y 的架构是什么？
3. Z 功能的验收标准是什么？
```

**典型工作流：**
1. 定期将会议记录 (MD)、设计文档 (DOCX) 导入
2. LLM 生成结构化的 entity pages (feature names, team members) 和 concept pages (architecture patterns)
3. Chrome Web Clipper 一键捕获网页参考
4. Git 仓库管理 wiki/ 目录，团队成员可 PR 更新

### 1.4 场景三：学术文献管理

**适用人群：** 研究生、文献综述作者

**典型工作流：**
1. 批量导入 ArXiv 论文 PDF
2. folderContext 传递分类信息（如 `"papers > NLP > transformers"`）
3. LLM 自动生成：
   - source summary (作者、年份、方法、结果)
   - entity pages (数据集名称、模型架构)
   - concept pages (技术方法、理论)
   - 论文间的交叉引用
4. 在 Graph View 中发现不同论文间的方法关联

### 4.5 MCP 高级玩法

**链式调用模式：**

```
1. Claude 收到用户问题:
   "我的 wiki 中关于强化学习的知识有什么缺口？"

2. Claude 调用 llm_wiki_graph 获取知识图谱
   分析低 cohesion 社区 → 发现 "RL + NLP" 子图稀疏

3. Claude 调用 llm_wiki_search 查询 "reinforcement learning nlp"
   确认现有页面覆盖范围

4. Claude 调用 llm_wiki_chat (mode: "deep")
   让 LLM Wiki Agent 分析知识缺口并建议研究方向

5. Claude 调用 llm_wiki_rescan_sources
   触发源文件重新扫描，确保所有论文都已处理
```

**安全模型：**
- MCP Server 只读默认（`PermissionPolicy::api_default()` in `permissions.rs`）
- Shell 命令需要显式审批
- 工作区文件操作限制在 `agent-workspace/` 内
- Token 认证 + `127.0.0.1` 绑定

## 5. RAG 增强方案

### 5.1 LLM Wiki 作为 RAG 的"预编译缓存层"

传统 RAG 管道可以前置 LLM Wiki 作为预处理步骤：

```
文档 (PDF/DOCX/MD)
    │
    ▼
┌──────────────────────────┐
│  LLM Wiki Ingest 管线     │  ← 预编译阶段
│  → structured wiki pages  │
│  → entity/concept graph   │
│  → cross-references       │
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│  传统 RAG 检索            │  ← 查询阶段
│  → 检索 wiki pages 而非   │
│    原始文档片段            │
│  → 更高质量、更准确的      │
│    结构化知识              │
└──────────────────────────┘
```

**优势：**
- Wiki 页面已经过 LLM 摘要、去噪和组织
- 交叉引用提供比原始 chunk 更丰富的上下文
- 矛盾已被标记，避免 RAG 返回冲突信息

### 5.2 混合检索：三路融合

LLM Wiki 内置的检索管线本身就是一种先进的混合检索实现：

```
输入查询
    │
    ├──→ 路 1: Tokenized Search (BM25-like)
    │     ├── 中文: CJK bigram
    │     ├── 英文: 停用词过滤 + 词干匹配
    │     └── title match bonus (+10)
    │
    ├──→ 路 2: Vector Semantic Search (LanceDB)
    │     ├── text-chunker (递归 Markdown 分块)
    │     ├── embedding fetching (OpenAI-compatible)
    │     └── ANN → cosine similarity → top-K
    │
    └──→ 路 3: Graph Expansion
          ├── 4-Signal Relevance Model
          └── 2-hop traversal with decay
```

### 5.3 与 LangChain / LlamaIndex 集成

```python
# 方案：通过 HTTP API 将 LLM Wiki 作为 LangChain 的 Retriever

from langchain.retrievers import BaseRetriever
import requests

class LlmWikiRetriever(BaseRetriever):
    """LLM Wiki as a LangChain retriever"""

    base_url: str = "http://127.0.0.1:19828/api/v1"
    api_token: str = ""
    project: str = "current"
    top_k: int = 10

    def _get_relevant_documents(self, query: str):
        # 使用混合搜索
        resp = requests.post(
            f"{self.base_url}/projects/{self.project}/search",
            headers={"Authorization": f"***",
                     "Content-Type": "application/json"},
            json={"query": query, "topK": self.top_k, "includeContent": True}
        )
        results = resp.json()["results"]

        from langchain.schema import Document
        docs = []
        for item in results:
            docs.append(Document(
                page_content=item.get("content", item.get("snippet", "")),
                metadata={
                    "title": item["title"],
                    "path": item["path"],
                    "score": item["score"],
                    "source": "llm-wiki"
                }
            ))
        return docs

# 使用
retriever = LlmWikiRetriever(api_token="***)
from langchain.chains import RetrievalQA

qa = RetrievalQA.from_chain_type(
    llm=your_llm,
    chain_type="stuff",
    retriever=retriever
)

answer = qa.run("What is the transformer architecture?")
```

## 6. Deep Research 实战

### 6.1 触发方式

```

方式三: 手动触发
  → Research 面板 → "New Research"
  → 输入主题
  → LLM 生成优化的搜索词
```

### 6.3 研究管线流程

```
Deep Research 流程 (src/lib/deep-research.ts, 373 行):

1. 搜索源收集 (collectResearchSources)
   ├── 对每个搜索词执行 web search (max 20 条结果)
   ├── 可选: AnyTxt 本地文件搜索
   └── 去重 (seenUrls set)

2. LLM 合成 (synthesizeResearch)
   ├── 加载所有搜索结果的内容
   ├── 加载现有 wiki 页面作为上下文
   └── LLM 生成综合研究页面:
       ├── 摘要
       ├── 关键发现
       ├── 与现有 wiki 的关系
       └── 进一步研究建议

3. 自动 Ingest
   ├── 研究结果保存为 wiki/queries/research-{slug}-{date}.md
   └── 自动触发 ingest，提取新实体/概念进入 wiki

4. 并发控制
   └── 最多 3 个并发研究任务
```

### 6.4 研究主题优化

```typescript
//             Processing: RLHF, text-based games, and language agents"
//   搜索词:
```

## 7. Review 系统最佳实践

### 7.1 Review 项类型

LLM 在 ingest 过程中生成的 Review 项（`src/lib/review-utils.ts`）：

| Type | 含义 | 建议处理 |
|------|------|---------|
| `missing-page` | 引用了不存在的实体/概念页面 | Create Page / Deep Research |
| `duplicate` | 检测到两个页面描述同一实体 | Review → Merge via dedup |
| `contradiction` | 两个源对同一事实有矛盾说法 | Confirm → 标记在页面 frontmatter |
| `confirm` | LLM 不确定的推断，需人工确认 | Confirm / Skip |
| `suggestion` | LLM 建议的 Wiki 改进 | Skip / Create Page |

### 7.2 异步工作流

```
Ingest 阶段:
  LLM 生成 Review items → 存入 .llm-wiki/reviews.json
  └── 不阻塞 ingest 管线

用户空闲时:
  → 打开 Review 面板
  → 逐条处理: Create Page / Deep Research / Skip / Confirm
  → 已处理的条目标记 resolved

定期清理:
  → 运行 Lint 检查 Review 相关的问题
  → 定期清理已处理的 Review items
```

### 7.3 Pre-generated Search Queries

LLM 在生成 Review item 时同时生成优化的搜索词（`src/lib/review-create-page.ts`）：

## 9. Skill 系统扩展

### 9.1 SKILL.md 格式

LLM Wiki 的 Skill 系统允许为 Agent 注入自定义指令（`src-tauri/src/agent/skills.rs`）：

```markdown
---
name: code-review
description: "Review code changes for security and style issues"
version: 1.0.0
---

# Code Review Skill

## When to use
When the user asks to review code or check for security issues.

## Instructions
1. Read the file provided by the user
2. Check for:
   - SQL injection vulnerabilities
   - Hardcoded secrets
   - Input validation issues
3. Report findings with severity and suggested fixes

## Tool policy
- Use `workspace.read` to read source files
- Use `workspace.write` to create review reports
```

**存放位置：**
- 项目级: `.llm-wiki/skills/{skill-name}/SKILL.md`
- 用户级: `~/.llm-wiki/skills/{skill-name}/SKILL.md`

### 9.2 Skill 加载与使用

```typescript
// src-tauri/src/agent/skills.rs 的扫描逻辑：
//   扫描项目 .llm-wiki/skills/ 和用户 ~/.llm-wiki/skills/
```

### 9.3 Agent User Input Forms

Skill 可以定义动态表单，让 Agent 在运行时向用户请求结构化输入：

```yaml
# SKILL.md 中的 user_input 声明
user_input:
  fields:
    - name: target_env
      type: select
      label: "Target environment"
      options: ["staging", "production"]
      required: true
    - name: release_notes
      type: text
      label: "Release notes"
      required: false
```

**约束**（`src-tauri/src/agent/runtime.rs`）：
- `MAX_USER_INPUT_FIELDS = 12`
- `MAX_USER_INPUT_OPTIONS = 8`
- `MAX_USER_INPUT_TEXT_CHARS = 400`

### 9.4 Workspace 文件生成

Agent 可以将生成的文件输出到 `agent-workspace/`：

```
my-wiki/
└── agent-workspace/

    └── outputs/          # Agent 的其他输出
```

**预览支持**（`src-tauri/src/agent/workspace.rs`）：
