## 1. 执行摘要

本报告为工业交换机自动配置场景设计了一个**四层意图驱动架构**，核心思路是：

> **不直接从"需求 → 命令"，而是通过一个厂商无关的"配置意图中间表示（Intent IR）"来解耦方案设计和命令生成。**

这个架构的关键创新点：

| 你当前方案的痛点 | 本方案的解决策略 |
|---|---|
| 复杂场景方案设计错误率高 | **知识图谱 + 设计模式库 + RAG 三重知识注入**，让 LLM 有据可依 |
| 华为多版本命令差异大 | **Intent IR 中间层解耦**，命令生成时按具体版本检索命令手册 |
| 设备太多，命令数据库太大 | **分层检索 + 按需加载**：不存全量命令，只存 IR→命令模板的映射规则 |
| 缺乏专家经验参考 | **KG 提取论文的自动化方法**：从设备手册中自动提取结构化知识图谱 |

学术验证：该架构的核心思路已被多篇顶会论文验证——
- **INTA** (2025): Intent-Based Translation, 98.15%配置准确率
- **RAG Multi-Agent** (2026): 94.4% 成功率，任务分解+闭环验证
- **KG for Ethernet Switch** (2026): 从交换机手册提取知识图谱，正确率0.97-0.99

工程基础：Netmiko (4,224 stars) 已原生支持华为设备（huawei, huawei_vrp, huawei_vrpv8），可作为 SSH 下发层。

## 2. 现有方案问题诊断

### 2.1 你当前方案的架构

```
拓扑+需求 → LLM理解 → 方案设计 → 命令翻译 → SSH下发
```

### 2.2 根本问题分析

**问题一：LLM 方案设计缺乏"脚手架"**

你让 LLM 直接从需求跳到方案，相当于让一个没有参考资料的工程师凭空设计。LLM 的预训练知识对通用网络概念有覆盖，但对：
- 工业场景的特殊约束（确定性时延、冗余环网协议如 MRP/HSR/PRP、IEEE 802.1Qbv 时间感知整形）
- 华为设备特有的实现方式（如 SEP 智能以太网保护 vs 标准 ERPS）
- 特定行业的合规要求（电力 IEC 61850、工厂自动化的 Profinet/CC-Link IE 要求）

这些领域知识在 LLM 预训练语料中密度极低。你需要一个**可检索的外部知识库**充当"参考手册"。

**问题二：多版本命令差异的根本解法不是"存所有命令"**

华为 VRP 平台经历了 V3→V5→V8 的演进，加上 CloudEngine 系列运行 VRPv8 和 VRPv8 的不同维护版本，同一功能在不同版本上的 CLI 命令确实存在差异。例如：
- VRPv5: `display current-configuration`
- VRPv8: `display current-configuration` (部分输出格式变化)
- CE 系列: 支持 YANG/NETCONF，CLI 和 YANG 路径并存

但核心发现是：**大部分命令的语法结构是稳定的，真正变化的参数约占总量的 15-20%。** 所以不需要存全量命令数据库，而是应该：
1. 定义意图→命令的**映射规则**（跨版本通用）
2. 仅对差异部分维护**版本差异映射表**（轻量）

### 4.2 核心创新点详解

#### 创新 1: Intent IR（配置意图中间表示）

这是整个架构的核心解耦层。Intent IR 是一个**厂商无关、版本无关**的结构化配置意图描述语言。

**Intent IR 示例** (JSON/YAML格式):
```yaml
# 配置意图：在三层交换机上创建一个VLAN接口
intents:
  - id: "vlan-100-create"
    type: vlan_create
    parameters:
      vlan_id: 100
      description: "生产车间A网络"
    context:
      device_role: distribution_switch
      
  - id: "vlanif-100-ip"
    type: l3_interface_ip
    depends_on: ["vlan-100-create"]
    parameters:
      interface: Vlanif100
      ip_address: "192.168.100.1/24"
      vrrp_group: 1
      vrrp_priority: 120
      
  - id: "port-trunk-uplink"
    type: port_trunk_allow_vlan
    parameters:
      interface: "10GE1/0/1"
      mode: trunk
      allowed_vlans: [100, 200, 300]
      native_vlan: 1
```

**Intent IR 的设计原则**：
1. **功能粒度**：每个 intent 对应一个原子配置操作（创建 VLAN、设 IP、配路由...）
2. **参数充足**：包含所有该操作需要的参数，不预设默认值
3. **依赖显式声明**：依赖关系明确标注（如 VLAN 必须先创建才能配 VLANIF）
4. **上下文绑定**：标注设备角色、拓扑位置等上下文信息

#### 创新 2: 四层知识注入

| 层次 | 名称 | 内容 | 构建方式 |
|---|---|---|---|
| L1 | 设计模式库 | 常见拓扑方案模板（环形、星形、双归等） | 专家梳理 + 少量标注 |
| L2 | 知识图谱 | 从设备手册自动提取的配置项→依赖→约束关系 | 复用KG论文的EEI Pipeline |
| L3 | RAG 文档库 | 华为官方配置命令手册、产品文档 | Chunk + Embedding + 向量检索 |
| L4 | 约束规则引擎 | 硬性约束（不能配冲突IP/VLAN/VPN），行业合规规则 | 规则引擎（非LLM） |

#### 创新 3: 轻量级版本差异处理

不存全量命令，而是：
1. 维护一个**基础命令模板库**（以 VRPv8 最新版为基准）
2. 维护**版本差异映射表**（JSON，仅约 200-500 条差异规则）
3. 下发时根据设备版本查询差异表，做局部修正

```json
{
  "version_diff": [
    {
      "intent_type": "stp_mode",
      "vrpv5": "stp mode {mode}",
      "vrpv8": "stp mode {mode}",
      "vrpv8_r22+": "stp mode {mode}\nstp pathcost-standard {standard}",
      "ce_series": "stp mode {mode}\nstp region-configuration"
    }
  ]
}
```

## 5. 详细组件设计

### 5.1 拓扑解析 Agent

**输入**: 
- 手绘/Visio 拓扑图 → 视觉模型解析
- CSV/Excel 设备清单
- 自然语言描述 "两台核心、六台接入，核心间40G堆叠..."

**处理流程**:
```
输入 → 视觉/文本解析 → 拓扑规范化 → 设备角色识别 → 生成拓扑图模型
```

**拓扑模型结构**:
```python
class TopologyModel:
    devices: List[Device]        # 设备节点
    links: List[Link]            # 物理连接
    roles: Dict[str, Role]       # 角色标注
    subnets: List[Subnet]        # 网段规划
    
class Device:
    model: str                   # "S5732-H48XUM2CC"
    role: DeviceRole             # CORE / DISTRIBUTION / ACCESS
    version: str                 # "VRPv8 R22"
    ports: List[Port]
    management_ip: str
```

### 5.2 方案设计 Agent（核心）

这是最复杂的 Agent，负责从拓扑+需求生成 Intent IR。

**工作流程**:

```
Step 1: 需求理解 ──→ LLM 解析自然语言需求，提取显式+隐式需求
                      ↓
Step 2: 拓扑分析 ──→ 分析拓扑模型，识别关键路径、冗余关系
                      ↓
Step 3: 知识检索 ──→ 并行查询知识图谱 + 设计模式库 + RAG
                      ↓                    ↓              ↓
                  约束规则        常见方案模板      配置命令手册
                      ↓                    ↓              ↓
Step 4: 方案生成 ──→ LLM 综合所有检索结果，生成 Intent IR 列表
                      ↓
Step 5: 方案验证 ──→ 约束规则引擎检查 + 拓扑一致性验证
                      ↓
Step 6: 修正迭代 ──→ 如发现问题，返回 Step 3 补充检索后重新生成
```

**关键：RAG 检索的具体实现**

使用混合检索策略：
```python
# 伪代码
def retrieve_knowledge(query: str, context: DesignContext):
    results = []
    
    # 1. 向量检索（语义相似）
    semantic_hits = vector_search(
        query=query, 
        index="switch_manuals",
        top_k=10
    )
    
    # 2. 关键词检索（精确匹配）
    keyword_hits = bm25_search(
        query=extract_keywords(query),
        index="switch_manuals",
        top_k=10
    )
    
    # 3. 知识图谱检索（结构化查询）
    kg_hits = kg_query(
        entity_type=context.device_role,
        relation="requires_config",
        top_k=10
    )
    
    # 4. 设计模式匹配
    pattern_hits = pattern_match(
        topology_type=context.topology_type,
        requirements=context.requirements
    )
    
    # 5. RRF 融合（Reciprocal Rank Fusion）
    fused = reciprocal_rank_fusion([
        semantic_hits, keyword_hits, kg_hits, pattern_hits
    ])
    
    return fused[:15]
```

### 5.3 命令生成层

#### 5.3.1 YANG/NETCONF 通道（优先）

对于支持 NETCONF 的设备（VRPv8+，CE 系列）：

```python
# Intent IR → YANG 路径映射
INTENT_TO_YANG = {
    "vlan_create": {
        "yang_path": "/huawei-vlan:vlans/vlan",
        "key": "vlan-id",
        "params_map": {
            "vlan_id": "vlan-id",
            "description": "description"
        }
    },
    "l3_interface_ip": {
        "yang_path": "/ietf-ip:interfaces/interface",
        "key": "name",
        "params_map": {
            "interface": "name",
            "ip_address": "ietf-ip:ipv4/address/ip",
            "vrrp_group": "huawei-vrrp:vrrp/vrrp-group/vrrp-id"
        }
    }
}

def intent_to_netconf(intent: IntentIR) -> NetconfRPC:
    if intent.type in YANG_SUPPORTED:
        mapping = INTENT_TO_YANG[intent.type]
        return build_edit_config_rpc(mapping, intent.parameters)
    else:
        return fallback_to_cli(intent)
```

### 5.5 配置下发层

```python
class DeploymentManager:
    """配置下发管理器"""
    
    def deploy(self, configs: List[DeviceConfig]) -> DeployResult:
        results = []
        
        for config in configs:
            # Step 1: 预检查
            pre_check = self.pre_check(config.device)
            if not pre_check.ok:
                results.append(DeployResult(
                    device=config.device,
                    status="pre_check_failed",
                    detail=pre_check.errors
                ))
                continue
            
            # Step 2: 生成当前配置快照
            snapshot = self.get_running_config(config.device)
            
            # Step 3: 差异对比
            diff = self.compute_diff(snapshot, config.target_config)
            if diff.is_empty():
                results.append(DeployResult(
                    device=config.device,
                    status="no_change_needed"
                ))
                continue
            
            # Step 4: 下发配置
            try:
                self.apply_config(config.device, diff)
                
                # Step 5: 下发后验证
                post_check = self.post_verify(config.device, config.intent)
                if post_check.ok:
                    results.append(DeployResult(
                        device=config.device,
                        status="success"
                    ))
                else:
                    # Step 6: 自动回滚
                    self.rollback(config.device, snapshot)
                    results.append(DeployResult(
                        device=config.device,
                        status="rolled_back",
                        detail=post_check.errors
                    ))
            except DeployError as e:
                self.rollback(config.device, snapshot)
                results.append(DeployResult(
                    device=config.device,
                    status="failed",
                    detail=str(e)
                ))
        
        return DeploySummary(results)
```

### 7.3 版本自适应命令生成

```python
class VersionAdaptiveCommandGenerator:
    """版本自适应命令生成器"""
    
    def __init__(self):
        self.diff_table = load_version_diff_table()
        self.template_engine = Jinja2Engine()
        self.rag_retriever = RAGRetriever()
    
    def generate(self, intent: IntentIR, device: Device) -> List[str]:
        # Step 1: 判断设备能力
        device_cap = self.detect_device_capability(device)
        
        # Step 2: YANG 优先路径
        if device_cap.yang_supported and intent.type in YANG_CAPABLE:
            return self._generate_yang(intent, device)
        
        # Step 3: CLI 模板路径
        cli_family = self._resolve_cli_family(device.model, device.version)
        
        # Step 3a: 尝试直接从版本差异表获取
        diff_entry = self.diff_table.lookup(intent.type)
        if diff_entry and cli_family in diff_entry.variants:
            template = diff_entry.variants[cli_family]
            return self.template_engine.render(template, intent.parameters)
        
        # Step 3b: 使用基础模板 + 差异修正
        base_template = self._get_base_template(intent.type)
        base_cmds = self.template_engine.render(base_template, intent.parameters)
        fixes = self.diff_table.get_fixes(intent.type, cli_family)
        if fixes:
            base_cmds = self._apply_fixes(base_cmds, fixes)
            return base_cmds
        
        # Step 3c: 模板都不命中，走 RAG + LLM 即时生成
        context = self.rag_retriever.retrieve(
            query=f"{intent.type} configuration {device.model} {device.version}",
            filter={"device_model": device.model}
        )
        return self.llm_generate(intent, context, device)
    
    def _resolve_cli_family(self, model: str, version: str) -> str:
        """根据设备型号+版本确定 CLI 家族"""
        # 先按型号模糊匹配
        for pattern, info in VERSION_PATTERNS.items():
            if fnmatch(model, pattern):
                return info["cli_family"]
        # 再按版本推断
        if version.startswith("VRPv5"):
            return "vrp5_default"
        elif version.startswith("VRPv8"):
            return "vrp8_r20+"  # 默认最新
        return "unknown"
```

## 总结

本方案的核心思想是**三层解耦**：

1. **逻辑与物理解耦**: 方案设计不直接面对设备命令，而是输出 Intent IR
2. **知识与推理解耦**: LLM 负责推理编排，专业知识由 KG+RAG+规则引擎提供
3. **意图与命令解耦**: Intent IR → (版本自适应) → CLI/YANG 命令

这套架构不仅解决你当前的两个核心痛点（复杂方案设计难 + 多版本命令差异），还具有很好的扩展性——未来支持多厂商设备时，只需添加新的命令生成通道，Intent IR 和方案设计层完全不需要改动。

**下一步建议**：
1. 先用一台华为 S5732 做实操验证，跑通 "拓扑→Intent IR→YANG下发" 的最小闭环
2. 从 3-5 本核心产品手册开始构建知识图谱（不必一开始覆盖所有设备）
3. 中期目标是覆盖 S5720/S5732/S5735 三个系列 + CE 系列交换机
