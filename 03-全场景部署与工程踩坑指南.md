# 报告三：【全场景部署与工程踩坑指南】

> **读者定位：** DevOps、SRE、需要部署到生产环境的工程师  
> **核心交付物：** 从零到生产可用的完整部署手册、性能调优、常见报错 TOP 10  
> **llm-wiki-compiler 版本：** v1.0.0

## 1. 环境要求与硬件评估

### 1.1 基础环境

| 项目 | 要求 | 说明 |
|------|------|------|
| **Node.js** | ≥ 24.0.0 | 推荐 volta 管理：`volta pin node@24` |
| **npm** | ≥ 10.x | 随 Node.js 24 内置 |
| **磁盘空间** | 最小 100MB + 项目空间 | 取决于 `sources/` 和 `wiki/` 大小 |
| **内存** | 最小 512MB | 仅 CLI 本身；LLM 推理发生在远端 |
| **操作系统** | macOS / Linux / Windows | 全平台支持 |

### 1.2 LLM Provider API Key 需求

| Provider | 环境变量 | 获取方式 |
|----------|---------|---------|
| **Anthropic** (默认) | `ANTHROPIC_API_KEY` 或 `ANTHROPIC_AUTH_TOKEN` | https://console.anthropic.com |
| **OpenAI 兼容** | `OPENAI_API_KEY` | https://platform.openai.com 或自建服务 |
| **Ollama** (本地) | `OLLAMA_HOST` (可选) | 无需 API Key，需本地安装 Ollama |
| **Claude Agent SDK** | 本地 Claude Code 登录 | `claude login` |
| **GitHub Copilot** | `GITHUB_TOKEN` | `gh auth refresh --scopes copilot` |
| **MiniMax** | `MINIMAX_API_KEY` | MiniMax 开放平台 |
| **Voyage AI** (嵌入) | `VOYAGE_API_KEY` | https://www.voyageai.com (anthropic/claude-agent 提供者的语义搜索) |

### 1.3 Token 消耗估算

| 操作 | 消耗来源 | 估算公式 |
|------|---------|---------|
| `ingest` | 无 LLM 调用 | 0 Token |
| `compile` (Phase 1) | 每个源文件的概念提取 | source_length × 1.5 (prompt + output) |
| `compile` (Phase 2) | 每个合并概念的页面生成 | ~5000 tokens/page (prompt + output) |
| `query` | 检索 + 生成答案 | ~3000 tokens/query |
| `run_eval --suite full` | LLM 评判引文支持度 | sample_size × ~1000 tokens/sample |

**示例计算（中等项目）：**

```
10 个源文件，平均 5000 字 → 10 × 7500 tokens = 75,000 tokens (Phase 1)
生成 15 个概念页面 → 15 × 5000 tokens = 75,000 tokens (Phase 2)
总计一次全量编译 ≈ 150,000 tokens ≈ 约 $0.45 (Claude Sonnet)
增量编译（1 个文件变更） ≈ 15,000 tokens ≈ $0.045
```

## 2. 三种部署模式

### 2.1 本地开发者模式（推荐起步）

```bash
# ① 安装 Node.js 24
# 使用 volta:
curl https://get.volta.sh | bash
volta install node@24

# 使用 nvm:
nvm install 24
nvm use 24

# ② 全局安装
npm install -g llm-wiki-compiler

# ③ 配置 Provider
export ANTHROPIC_API_KEY=sk-ant-...
# 或在项目目录创建 .env 文件

# ④ 快速开始
mkdir my-wiki && cd my-wiki
llmwiki quickstart https://en.wikipedia.org/wiki/Your_Topic
llmwiki view --open
```

### 2.2 Docker 容器化部署

```dockerfile
# Dockerfile
FROM node:24-alpine

RUN npm install -g llm-wiki-compiler

WORKDIR /wiki
VOLUME ["/wiki/sources", "/wiki/wiki", "/wiki/.llmwiki"]

ENV LLMWIKI_PROVIDER=anthropic
# API key 通过 docker run -e 或 secrets 注入

ENTRYPOINT ["llmwiki"]
```

```bash
# 构建
docker build -t llmwiki .

# 运行
docker run -it --rm \
  -v $(pwd)/sources:/wiki/sources \
  -v $(pwd)/wiki:/wiki/wiki \
  -e ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY \
  llmwiki compile
```

### 2.3 CI/CD 集成（GitHub Actions）

```yaml
# .github/workflows/wiki-quality.yml
name: Wiki Quality Gate

on:
  push:
    paths:
      - 'sources/**'
      - '.llmwiki/**'

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 24

      - run: npm install -g llm-wiki-compiler

      - name: Lint
        run: llmwiki lint

      - name: Eval (fast)
        run: llmwiki eval --suite fast

      - name: Recompile changed sources
        run: llmwiki compile
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}

      - name: Quality gate
        run: |
          # 如果 eval 健康分低于阈值，失败
          llmwiki eval --suite fast --record
```

## 3. 多 Provider 配置完整指南

### 3.1 Anthropic（默认）

```bash
export LLMWIKI_PROVIDER=anthropic
export ANTHROPIC_API_KEY=sk-ant-...
# 可选：自定义模型
export LLMWIKI_MODEL=claude-sonnet-4-6
# 可选：代理
export ANTHROPIC_BASE_URL=https://your-proxy.com
```

### 3.2 OpenAI 兼容

```bash
export LLMWIKI_PROVIDER=openai
export OPENAI_API_KEY=sk-...
# 自定义端点（任何 OpenAI 兼容服务）
export OPENAI_BASE_URL=https://api.openai.com/v1
# 可选：分离的 Embedding 端点
export OPENAI_EMBEDDINGS_BASE_URL=https://your-embedding-service.com/v1
# 自定义模型
export LLMWIKI_MODEL=gpt-4o
export LLMWIKI_EMBEDDING_MODEL=text-embedding-3-large
```

### 3.3 Ollama 本地（完全离线）

```bash
# 先确保 Ollama 已安装并运行
ollama pull llama3.1
ollama pull nomic-embed-text    # 用于语义搜索

export LLMWIKI_PROVIDER=ollama
export OLLAMA_HOST=http://localhost:11434/v1
export LLMWIKI_MODEL=llama3.1
# 可选：增大超时（本地推理慢）
export OLLAMA_TIMEOUT_MS=600000   # 10 分钟
```

### 3.4 Claude Agent SDK（复用本地 Claude Code 登录）

```bash
# 需要先安装 Claude Code 并登录
npm install -g @anthropic-ai/claude-code
claude login

export LLMWIKI_PROVIDER=claude-agent
# 无需 API Key——使用本地登录凭据
```

### 3.5 GitHub Copilot

```bash
# 需要 GitHub Copilot 订阅
gh auth refresh --scopes copilot
export LLMWIKI_PROVIDER=copilot
export GITHUB_TOKEN=$(gh auth token)
```

### 3.6 MiniMax

```bash
export LLMWIKI_PROVIDER=minimax
export MINIMAX_API_KEY=your_key
```

## 4. 编译性能调优

### 4.1 并行度控制

```bash
# 默认并发数：5
llmwiki compile

# 提高并发（更多并行 LLM 调用 → 更快但更容易触发 rate limit）
llmwiki compile --concurrency 10

# 通过环境变量
export LLMWIKI_COMPILE_CONCURRENCY=10

# 上限：50（超过会被 clamp 并警告）
```

**推荐值：**
| 场景 | 推荐并发 | 原因 |
|------|:---:|------|
| Anthropic (Tier 4) | 8-12 | API 支持高并发 |
| OpenAI (Tier 5) | 10-20 | 高并发配额 |
| Ollama 本地 | 2-4 | 本地 GPU 限制 |
| Copilot | 3-5 | 共享配额 |
| 大规模编译（100+ 文件） | 15-20 | 充分利用 wall-clock |

### 4.2 增量编译策略

llm-wiki-compiler 的核心性能优化是**基于 SHA-256 的增量编译**：

```bash
# 首次编译：处理所有源文件（全量）
llmwiki compile

# 后续编译：只处理新增/变更的文件（增量）
# 状态保存在 .llmwiki/state.json
llmwiki compile    # 自动增量！

# 查看哪些文件会触发重新编译
llmwiki status
```

**增量编译的节省效果：** 当 `sources/` 有 100 个文件，只修改其中 2 个时，Token 消耗仅为全量编译的 ~2%。

### 4.3 Prompt Budget 调节

```bash
# 大型概念（多源合并）可能超过默认 prompt budget 200K chars
# 提高上限（适用于大 context 模型如 Gemini 2M / GPT-4-128K）
export LLMWIKI_PROMPT_BUDGET_CHARS=400000

# 降低上限（适用于小 context 本地模型）
export LLMWIKI_PROMPT_BUDGET_CHARS=50000
```

### 4.4 Embedding 批处理优化

```bash
# 增大批处理大小（减少 API 往返次数）
export LLMWIKI_EMBED_BATCH_SIZE=256

# 严格模式：embedding 失败时退出（CI 中使用）
export LLMWIKI_EMBED_STRICT=1
```

## 5. 常见报错 TOP 10 与解决方案

### ① `ProviderUnavailableError: No API key found`

```
Error: No API key found for provider "anthropic"
```

**原因：** 未设置 provider 的 API key  
**解决：**
```bash
export ANTHROPIC_API_KEY=sk-ant-...
# 或创建 .env 文件在项目根目录
```

### ② `Node.js version mismatch`

```
Error: llm-wiki-compiler requires Node.js >= 24
```

**原因：** Node.js 版本过低  
**解决：**
```bash
node --version  # 检查当前版本
volta install node@24
# 或 nvm install 24 && nvm use 24
```

### ③ `Embedding endpoint not available`

```
Warning: embedding endpoint not available — falling back to lexical ranking
```

**原因：** 未配置嵌入提供者（Voyage AI / OpenAI embeddings）  
**解决（二选一）：**
```bash
# 方案 A：配置 Voyage AI（Anthropic 提供者下）
export VOYAGE_API_KEY=pa-...

# 方案 B：接受降级——lexical fallback 仍能工作
# 只是检索质量略低于语义搜索
```

### ④ `Source too large — content truncated`

```
Warning: source too large (250000 chars), truncated to 100000
```

**原因：** 源文件超过 `MAX_SOURCE_CHARS` (100,000 字符)  
**解决：**
- 拆分大文件为多个小文件
- 该限制是硬编码的，无法通过环境变量调整
- 合并概念页面的上限是 `GENERATED_PAGE_MAX_CHARS` (500,000 字符)

### ⑤ `Review policy config invalid — compile aborted`

```
Error: review policy config invalid — compile aborted
```

**原因：** `.llmwiki/config.json` 中的审查策略配置格式错误  
**解决：**
```bash
# 检查配置
cat .llmwiki/config.json
# 修正 JSON 格式错误
# 或删除配置文件恢复默认策略
rm .llmwiki/config.json
```
**设计原则：fail-closed**——无效配置拒绝编译，而非静默禁用审查

### ⑥ `Could not acquire .llmwiki/lock`

```
Error: Could not acquire .llmwiki/lock — another compile is in progress.
```

**原因：** 另一个编译进程正在运行，或上次崩溃未清理锁  
**解决：**
```bash
# 等待正在运行的编译完成
# 或强制清理（确认没有其他进程在编译！）
rm .llmwiki/lock
# 然后运行 recover 恢复可能的不一致状态
llmwiki recover
```

### ⑦ `Profile validation failed`

```
Error: profile validation failed — <具体错误>
```

**原因：** `.llmwiki/profile.json` 不符合 schema  
**解决：**
```bash
# 查看具体验证错误
llmwiki profile validate

# 重置为默认 profile
rm .llmwiki/profile.json
```

### ⑧ `OKF import collision`

```
Skipped page "xxx" — collides with existing page
```

**原因：** OKF 导入的页面与已有 Wiki 页面 slug 冲突  
**解决：**
```bash
# 用 --dry-run 预览冲突
llmwiki import --okf ./bundle --dry-run

# 手动重命名冲突页面后再导入
# 或使用 --trusted 强制覆盖（谨慎！）
llmwiki import --okf ./bundle --trusted
```

### ⑨ `Stale page repair failed`

```
Error: stale page repair failed for "bgp-path-selection"
```

**原因：** `llmwiki refresh --stale` 无法修复过时页面  
**解决：**
```bash
# 使用 --dry-run 预览修复计划
llmwiki refresh --stale --dry-run

# 手动检查源文件是否存在
ls sources/

# 如果源文件已被删除，接受孤儿页面状态
# 或重新创建源文件后重新编译
llmwiki compile
```

### ⑩ `Viewer port already in use`

```
Error: listen EADDRINUSE: address already in use :::3001
```

**原因：** 查看器端口被占用  
**解决：**
```bash
# 使用不同端口
llmwiki view --port 3002

# 或终止占用端口的进程
lsof -i :3001
kill -9 <PID>
```

## 7. 监控与运维

### 7.1 健康评分

```bash
# 快速评估（无 LLM 调用）
llmwiki eval --suite fast

# 完整评估（含 LLM 评判引文支持度）
llmwiki eval --suite full

# 记录到历史
llmwiki eval --suite fast --record
```

**评估指标：**
| 指标 | 含义 | 理想值 |
|------|------|:---:|
| 健康评分 | 综合质量分数 | ≥ 80 |
| 引文覆盖率 | 有引文支撑的段落比例 | ≥ 80% |
| 图健康 | wikilink 完整性 | 100% |
| 源文件利用率 | 所有源文件都被引用 | 100% |

### 7.2 自动化质量检查

```bash
# 运行 lint
llmwiki lint
# 检查项：broken wikilinks, orphans, duplicates, empty pages, broken citations

# 查看项目状态
llmwiki status
# 输出：页面数、源文件数、上次编译时间、stale/orphaned 页面、待处理变更

# 查看推荐的下一个操作
llmwiki next
```

### 7.3 日志审计

```bash
# 查看活动日志
cat log.md

# 日志格式：
# ## [2026-07-12] ingest | rfc4271-summary.md
# ## [2026-07-12] compile | 3 sources → 5 pages
```
